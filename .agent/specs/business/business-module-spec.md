# 业务域划分规范（Business Module Specification）

> **用途：** 定义模块化单体中业务域的划分原则和通用规则  
> **适用场景：** 任何需要划分业务域的项目  
> **复用性：** ⭐⭐⭐⭐ 所有模块化单体项目可参考

---

## 一、什么是业务域

**业务域（Business Domain）** 是具有独立业务概念、数据集合和业务能力的逻辑单元。

**特征：**
- 独立的业务概念（如用户、商品、订单）
- 独立的数据集合（专属的数据库表）
- 独立的业务能力（能独立完成某项功能）
- 低耦合高内聚（与其他域交互少，内部关联紧密）

---

## 二、如何识别业务域

### 2.1 判断标准

| 标准 | 说明 | 示例 |
|------|------|------|
| **独立概念** | 有明确的业务含义 | 用户、商品、订单 |
| **独立数据** | 有专属的数据库表 | user_info、product_info、order_info |
| **独立能力** | 能独立完成业务功能 | 用户注册、商品上架、订单创建 |
| **低耦合** | 与其他域交互较少 | 用户域被多个域调用，但很少调用其他域 |
| **高内聚** | 内部功能紧密相关 | 订单域包含订单创建、支付、取消等功能 |

---

### 2.2 常见业务域示例

#### 电商系统

| 业务域 | 职责 | 核心功能 |
|--------|------|---------|
| **用户域（user）** | 用户管理、认证授权 | 注册、登录、个人信息、权限 |
| **商品域（product）** | 商品管理、库存管理 | SPU/SKU、上下架、库存 |
| **订单域（order）** | 订单管理、交易流程 | 创建订单、支付、取消、售后 |
| **支付域（payment）** | 支付处理 | 发起支付、回调、退款 |
| **营销域（promotion）** | 营销活动 | 优惠券、活动、积分 |
| **物流域（logistics）** | 物流配送 | 发货、配送、签收 |

#### SaaS 系统

| 业务域 | 职责 | 核心功能 |
|--------|------|---------|
| **租户域（tenant）** | 租户管理 | 租户注册、配置、订阅 |
| **用户域（user）** | 用户管理 | 用户邀请、角色、权限 |
| **产品域（product）** | 产品功能 | 功能模块、套餐 |
| **账单域（billing）** | 计费管理 | 用量统计、账单生成、扣费 |

## 三、业务域之间的交互原则

### 3.1 交互方式

**唯一允许的方式：通过 Client 接口调用**

```
order-application → user-client（接口） → user-application（实现）
```

**禁止的方式：**
- ❌ 直接访问其他域的 Mapper/DAO
- ❌ 直接调用其他域的 ServiceImpl
- ❌ 直接查询其他域的数据库表

---

### 3.2 交互场景

#### 同步调用（核心流程）

**场景：** 订单创建时需要获取用户信息和商品信息

**流程：**
1. OrderApplication 调用 UserClient.getUserById()
2. OrderApplication 调用 ProductClient.getProductById()
3. OrderApplication 创建订单

**特点：**
- 强一致性要求
- 实时返回结果
- 失败则整个流程失败

---

#### 异步通知（非核心流程）

**场景：** 订单创建成功后发送通知、更新积分

**流程：**
1. OrderApplication 创建订单
2. OrderApplication 发布 OrderCreatedEvent 事件
3. NotificationConsumer 监听事件，发送通知
4. PointsConsumer 监听事件，更新积分

**特点：**
- 最终一致性
- 解耦核心流程
- 失败不影响主流程

---

### 3.3 禁止的交互

#### 禁止跨域 JOIN 查询

**错误示例：**
```sql
SELECT o.id, u.username 
FROM order_info o
JOIN user_info u ON o.user_id = u.id  -- ❌ 禁止
```

**正确做法：**
1. 查询订单表获取 user_id
2. 调用 UserClient.getUserById(user_id) 获取用户名
3. 在应用层组装结果

---

#### 禁止跨域事务

**错误示例：**
```java
@Transactional
public void createOrderWithUser() {
    orderGateway.create(order);      // 写订单表
    userGateway.updateLastOrderTime(); // 写用户表 ❌ 禁止
}
```

**正确做法：**
1. 本地事务创建订单
2. 发布领域事件
3. 异步更新用户信息（最终一致性）

---

#### 禁止循环依赖

**错误示例：**
- order-application → user-client
- user-application → order-client

**解决：**
- 重新审视业务边界
- 提取公共逻辑到 common 模块
- 引入事件机制解耦

---

## 四、业务域内部的子模块组织

每个业务域包含 4 个标准子模块：

```
{business}/                          # 业务域父模块（pom 打包）
├── {business}-client/               # 接口契约层（对外唯一出口）
├── {business}-domain/               # 领域模型层（核心业务逻辑）
├── {business}-application/          # 应用服务层（业务流程编排）
└── {business}-adaptor/              # API 接入层（HTTP 接口）
```

**各层职责简述：**
- **Client：** 定义 Service 接口和 DTO
- **Domain：** 实现核心业务逻辑和领域模型
- **Application：** 协调 Domain 层，实现业务流程
- **Adaptor：** 接收 HTTP 请求，调用 Application 层

**详细规范参考：** `architecture/` 目录下的分层 Spec

## 五、数据库层面的业务隔离

### 5.1 表命名规范

**规则：表名加业务域前缀**

| 业务域 | 表前缀 | 示例 |
|--------|--------|------|
| 用户域 | `user_` | user_info、user_role |
| 商品域 | `product_` | product_info、product_sku |
| 订单域 | `order_` | order_info、order_item |

**好处：**
- 一眼看出表属于哪个业务域
- 未来分库时便于迁移
- 避免表名冲突

---

### 5.2 数据访问限制

**单体阶段：**
- ✅ 可以共用一个数据库实例
- ✅ 表必须按业务域前缀区分
- ❌ 禁止跨域 JOIN
- ❌ 禁止跨域事务

**微服务阶段：**
- ✅ 每个业务域独立数据库
- ✅ 表前缀可以去掉（已分库）

---

## 六、业务域划分的原则

### 6.1 单一职责

**原则：** 一个业务域只负责一个核心业务概念

**好的划分：**
- 用户域：只管理用户相关信息
- 订单域：只管理订单相关流程

**不好的划分：**
- 订单域同时管理用户、商品、支付（职责过多）

---

### 6.2 高内聚低耦合

**高内聚：** 业务域内部的功能紧密相关

**示例：** 订单域包含订单创建、支付、取消、售后，都是订单生命周期的一部分

**低耦合：** 业务域之间的依赖尽量少

**示例：** 用户域很少调用其他域，但被订单域、营销域调用

---

### 6.3 稳定优先

**原则：** 变化频率低的业务优先划分为独立域

**稳定的业务域：**
- 用户域（用户管理相对稳定）
- 商品域（商品模型相对稳定）

**易变的业务域：**
- 营销域（活动规则经常变化）→ 可以考虑合并或灵活设计

---

### 6.4 团队结构对齐

**康威定律：** 系统设计受限于组织沟通结构

**建议：**
- 一个业务域由一个小团队负责（2-4 人）
- 业务域边界与团队职责对齐
- 减少跨团队协作成本

---

## 七、业务域演化的策略

### 7.1 初期简化

**初创阶段：**
- 业务域数量控制在 3-5 个
- 不必过度拆分
- 快速验证业务模式

**示例：** 电商 MVP 版本
- 用户域
- 商品域
- 订单域

---

### 7.2 逐步细化

**业务发展后：**
- 根据业务复杂度拆分新域
- 保持架构一致性
- 避免大规模重构

**示例：** 电商成熟期
- 从订单域拆分出支付域
- 新增营销域
- 新增物流域

---

### 7.3 合并策略

**何时合并：**
- 两个域总是同时修改
- 域之间交互过于频繁
- 团队规模缩小

**如何合并：**
- 保留一个域的名称
- 迁移代码和数据
- 更新依赖关系

## 九、快速检查清单

划分业务域时，确认以下事项：

- [ ] 业务域有独立的业务概念
- [ ] 业务域有专属的数据库表
- [ ] 业务域能独立完成某项功能
- [ ] 业务域之间通过接口调用
- [ ] 没有跨域 JOIN 查询
- [ ] 没有跨域事务
- [ ] 表名加了业务域前缀
- [ ] 业务域数量适中（3-5 个为宜）
- [ ] 业务域边界清晰，无重叠

---

## 十、常见错误

### 错误 1：业务域划分过细

**表现：** 一个简单系统拆分成 10+ 个业务域

**后果：**
- 开发效率降低
- 跨域调用复杂
- 维护成本高

**解决：** 合并相关性强的域，控制在 3-5 个

---

### 错误 2：业务域划分过粗

**表现：** 所有业务逻辑放在一个域中

**后果：**
- 代码耦合严重
- 难以理解和维护
- 未来拆分困难

**解决：** 按业务概念拆分为多个域

---

### 错误 3：跨域直接访问数据库

**表现：** order-application 直接查询 user_info 表

**后果：**
- 违反封装原则
- 未来拆分微服务时需要大规模重构

**解决：** 通过 UserClient 接口调用

---

### 错误 4：循环依赖

**表现：** A 域调用 B 域，B 域又调用 A 域

**后果：**
- Maven 构建失败
- 业务逻辑混乱

**解决：**
- 重新审视业务边界
- 提取公共逻辑到 common
- 引入事件机制解耦
